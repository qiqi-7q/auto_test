Pytest 是 Python 官方的单元测试框架，用于自动化执行测试代码。它通过扫描指定目录下的文件，自动识别符合命名规范的测试函数并执行，最后收集所有断言结果生成测试报告。相比 Python 内置的 unittest ，Pytest 更加轻量、简洁，断言失败时能自动显示详细的对比信息。
Pytest 有一个强大的自动发现规则，它会递归遍历目录，找到所有符合命名模式的文件、类和函数。文件名必须以 test_ 开头或以 _test.py 结尾；类名必须以大写 Test 开头（但不能是 Test__init__ ）；函数名必须以 test_ 开头。只要遵循这些命名约定，Pytest 就能自动收集并执行这些测试，无需额外配置。
每个测试方法执行前后，可能需要初始化一些资源或清理环境。Pytest 提供了 setup_method 和 teardown_method 来处理这个需求。前者会在每个测试方法之前执行，后者会在每个测试方法之后执行。在你的代码中， setup_method 用于创建 APIClient 实例， teardown_method 用于重置 token，这样可以确保每个测试都独立运行，互不影响。
参数化测试：参数化允许你用一组不同的参数多次执行同一个测试函数。通过 @pytest.mark.parametrize 装饰器，你可以定义多组参数，Pytest 会为每一组参数生成一个独立的测试用例。例如 @pytest.mark.parametrize("vehicle_id", [1000, 1001]) 会让这个测试函数执行两次，一次用 1000，一次用 1001，这样就不需要写两个重复的测试函数了。
有时候某些测试暂时无法执行，比如功能未开发完成、需要特定环境、或者已知存在问题。 @pytest.mark.skip 装饰器可以跳过这个测试，并在输出中显示跳过原因。如果想根据条件跳过，比如判断 Python 版本或某个配置项是否满足，可以使用 @pytest.mark.skipif ，只有条件为 False 时才跳过测试。
Pytest 提供了丰富的命令行选项来控制测试执行方式。 -v 会显示每个测试的名称和结果，方便查看详细输出； -s 会显示测试中的 print 输出； -x 会在遇到第一个失败时停止运行； -k 可以按名称筛选测试，比如 -k "valid" 会只运行名称包含 "valid" 的测试； --tb=short 会简化错误回溯信息，让失败原因更易读。
Pytest 使用 Python 原生的 assert 语句进行断言，而不是像 unittest 那样提供一堆 assertEqual 、 assertTrue 等方法。好处是代码更简洁，而且当断言失败时，Pytest 会自动分析比较的两边，分别显示实际值和期望值，帮助快速定位问题。比如 assert result == expected 失败时，会清晰显示两边各自的值。

遇到的问题：异步数据库（async SQLAlchemy）有一个死规矩：
👉 执行完 await db.commit () 之后，你绝对不能再碰 db_task.xxx 这个对象！
👉 一碰，就会触发数据库偷偷查询 → 直接报错 MissingGreenlet
错误：
async def create_test_task(db: AsyncSession, task: schemas.TestTaskCreate):
    # 1. 创建任务
    db_task = models.TestTask(**task.dict())
    db.add(db_task)

    # 2. 提交数据库
    await db.commit()  

    # 3. 提交后，还去访问 db_task 的属性 → 报错！
    return schemas.TestTask(
        id=db_task.id,            # 报错！
        project=db_task.project,  # 报错！
        created_at=db_task.created_at  # 报错！
    )

正确：
# 正确！！！
async def create_test_task(db: AsyncSession, task: schemas.TestTaskCreate):
    # 1. 创建任务
    db_task = models.TestTask(**task.dict())
    db.add(db_task)

    # 2. flush 拿到 ID（不提交）
    await db.flush()
    task_id = db_task.id

    # 🔥 关键：所有数据全部提前拿好，不用 db_task
    return_data = {
        "id": task_id,
        "project": task.project,   # 用前端传过来的数据！
        "test_time": task.test_time,
        "vin_code": task.vin_code,
        "created_at": datetime.now()
    }

    # 3. 提交数据库
    await db.commit()

    # 4. 直接返回提前准备好的数据
    return schemas.TestTask(**return_data)
在异步 SQLAlchemy 环境中，执行await db.commit()后 ORM 实体对象会被标记为过期，直接访问对象属性会触发隐式数据库查询，进而抛出MissingGreenlet错误；因此本方案的解决思路为：在执行事务提交前提前提取所需数据，不再依赖提交后的实体对象，优先使用接口入参构造返回结果与同步数据，全程规避提交后读取过期对象的行为，以此彻底兼容异步数据库的运行规则，避免报错。

nginx ：对浏览器 → Nginx 是【服务端】   对后端 FastAPI → Nginx 是【客户端】

浏览器仅会主动发起请求访问 Nginx 的 IP 和端口，全程不知道后端真实地址；当 Nginx 识别到请求路径以 /api 开头时，会独立新建一条网络连接，以自身服务器作为客户端主动去请求后端服务，后端处理完成后将数据返回给 Nginx，再由 Nginx 把结果原路传回浏览器，整个后端转发过程由 Nginx 内部完成，对浏览器完全透明，浏览器不存在自动跳转、直连后端的行为。

nginx服务器存储前端代码，可以理解为访问nginx的ip+端口就是在访问前端界面(前端页面的访问地址就是nginx的ip+端口，事实上，前端代码只是存放在 Nginx 管理的服务器目录里，Nginx 只是负责把目录里的 html/js/css 静态文件返回给浏览器，前端代码本身没有独立端口、没有独立服务进程，它只是一堆静态文件，依附 Nginx 对外提供访问)

自己理解：浏览器作为用户客户端，仅通过 Nginx 的 IP+8080 端口访问前端页面；页面发起的所有接口请求，地址依旧是 Nginx 的 IP + 端口拼接 /api/xxx。当 Nginx 识别到请求匹配 /api 或 /static/ 规则时，它会切换身份：原本面向浏览器是被动等待连接的服务端，此刻主动充当客户端，新建网络连接去访问真实后端 10.192.183.117:8000，拿到后端返回的数据后再原路传回浏览器，整个转发过程浏览器完全感知不到后端真实地址。
